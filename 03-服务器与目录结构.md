# 03 服务器与目录结构

## 服务器规格（参考）

| 角色 | 数量 | 配置 | 说明 |
| --- | --- | --- | --- |
| **portal 主机**（Linux x86_64） | 1 | 8 vCPU / 16 GB / 系统盘 100 GB + 数据盘 2 TB SSD | 跑 Jenkins master、Nexus、Sentry、MinIO、Caddy、decrypt-proxy、Filebrowser，全部 docker-compose。 |
| Linux build agent | 1~N | 8 vCPU / 16 GB | Android/Linux 构建，可复用 portal 主机也可独立。 |
| macOS build agent | 1 | mac mini M2 / 16 GB | iOS/macOS 构建必须裸机。 |
| Windows build agent | 1 | 8 vCPU / 16 GB | Windows DLL/MSI/PDB 构建。 |

> 容量估算：日志 100 MB/设备/天 × 1000 活跃设备 × 90 天 ≈ **9 TB**。生产环境建议 portal 主机数据盘做 RAID10 + 异机房 MinIO mirror。

如果数据量上百 TB，把 **MinIO 拆出去单独 4 节点 EC 集群**，其他组件继续放 portal。

## 对象存储（MinIO）bucket 布局

固定 2 个桶，**对象 key 自带元信息**，不依赖数据库：

### sdk-logs（日志 / Dump）

客户端打包 zip → 向 `upload-token` 换 presigned URL → PUT 直传。命名规范见[06-日志Dump上传方案.md](06-日志Dump上传方案.md)。

```
s3://sdk-logs/
  {data_type}/           ← cdc | cds
    {platform}/          ← windows | mac | linux | android | ios
      {version}/         ← 1.0.0
        {yyyy-mm-dd}/    ← 上传日期（精度到天）
          {uid}/         ← 设备/用户唯一标识
            log/{HH-mm-ss}-{uuid}.zip
            dump/{HH-mm-ss}-{uuid}.zip
```

### sdk-packets（SDK 构建产物）

Jenkins 出包后通过 `upload.bat`/`upload.ps1` 写入，按产品 → 分支 → 平台 → 版本号 → 日期（精度到天）分层。Lifecycle 见 [08-保留与清理策略.md](08-保留与清理策略.md)。

```
s3://sdk-packets/
  {product}/             ← cdc | cds
    {branch}/            ← develop | hotfix | rc | staging | release
      {platform}/        ← windows | mac | linux | android | ios
        {version}/       ← 1.0.0
          {date}/         ← 20260616（精度到天，从 zip 文件名中的时间戳提取）
            cdc-win-1.0.0-develop.51+e2de945-20260616-183702.zip
```

**分支提取规则**（由 upload.ps1 从版本号解析）：
| 版本号模式 | 分支 |
|---|---|
| `*-develop.*` | `develop` |
| `*-hotfix.*` | `hotfix` |
| `*-rc.*` 或 `*-staging.*` | `rc` / `staging` |
| 其他（如 `1.0.0`） | `release` |

上传前先 HEAD 检查对象是否已存在，已存在则跳过（幂等），不会覆盖。

## Nexus 仓库布局

在 Nexus Web → Repositories 新建：

| 仓库名 | 类型 | 说明 |
| --- | --- | --- |
| `sdk-raw-dev`       | raw (hosted) | dev 通道 SDK 包，写入开放给 Jenkins CI 账号。 |
| `sdk-raw-staging`   | raw (hosted) | 提测通道，由 Jenkins "提测" stage 写入。 |
| `sdk-raw-release`   | raw (hosted) | 线上通道，仅 release pipeline 双人审批后写入。 |
| `sdk-maven-dev/-staging/-release` | maven2 (hosted) | Android `.aar` 走 Maven 协议，方便 Gradle 直接 `implementation`。 |
| `sdk-cocoapods-release` | raw + spec repo | iOS xcframework 用 raw 存包，配套一个 spec repo (Gitea 仓库) 暴露 podspec。 |
| `sdk-docker`        | docker (hosted) | 编译环境镜像、内部工具镜像。 |
| `sdk-group`         | group | 把上面所有仓库聚合，开发者只配一个 URL 就能拉到正确通道。 |

对象 Path 约定（raw 仓库内部）：
```
{product}/{platform}/{version}-{buildId}/
   sdk.zip
   sdk.zip.sha256
   symbols.tar.zst        # 可选, Sentry 已存就免
   sbom.json
   metadata.json          # git sha, jenkins url, signer, ts
   buildlog.txt
```

## Sentry 项目布局

- Organization：`mycompany`。
- Project 按产品 × 平台拆：`rtc-sdk-android` / `rtc-sdk-ios` / `rtc-sdk-win` / `rtc-sdk-linux`。
- 每次构建上传符号表 + 设置 `release = {product}@{version}-{buildId}`，Crash 自动归位。
- Sentry "Environment" 字段填 `dev / staging / release`，与 Nexus 通道一致。
