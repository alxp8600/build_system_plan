# 05 包与符号表管理（对应核心问题 2）

> 一句话：**通道分仓（dev / staging / release）+ 不可变版本号 + 元数据 manifest + Sentry 自动符号化**。每个 SDK 包都能反查到 git commit、构建机、构建参数、符号表、SBOM。

## 1. 通道（channel）模型

| Channel | 定义 | 仓库 | 保留 | 谁能写 |
| --- | --- | --- | --- | --- |
| `dev` | 任意分支 push 自动出，给开发自测 | `sdk-raw-dev` / `sdk-maven-dev` | 30 天，自动清理 | Jenkins CI 账号 |
| `staging` | 提测包，给 QA 验证 | `sdk-raw-staging` / `sdk-maven-staging` | 180 天 | Jenkins，**input 双人审批**后写入 |
| `release` | 线上发布包，对外交付 | `sdk-raw-release` / `sdk-maven-release` / `sdk-cocoapods-release` | 永久（含归档冷存储） | release-manager 角色审批后写入；**禁止覆盖**（Nexus 设 `writePolicy=ALLOW_ONCE`） |

> 三通道**严格只升级、不降级**：dev 包永远不可推到 staging/release；提测包升级线上必须重新跑 release pipeline。

## 2. 版本号

格式（SemVer + 元数据）：
```
{major}.{minor}.{patch}[-{prerelease}][+{build}]
```
- dev：`1.7.0-dev.{jenkins#}+{gitsha7}`
- rc/staging：`1.7.0-rc.{N}+{gitsha7}`
- release：`1.7.0`（必须有签名的 git tag `v1.7.0`，由 Jenkins 校验）

版本号由 `scripts-stub/version.sh` 在每个 stage 起始计算，写入：
- `dist/{plat}/metadata.json` 的 `version` 字段
- Sentry release 名
- Nexus 路径
- 通知卡片

## 3. 不可变与可追溯

- Nexus raw 仓全部设 **`writePolicy=ALLOW_ONCE`**（写一次就不能改/删，除 admin 手动开通道）。
- 每个版本对象路径里都带 **Jenkins BUILD_NUMBER**，防止"同一版本号重打"覆盖。
- `metadata.json` 必填：

## 4. 符号表

### 4.1 主存：Sentry Debug Files

构建末尾执行：
```
sentry-cli debug-files upload \
  --org mycompany --project rtc-sdk-{plat} \
  --include-sources \
  dist/{plat}/symbols/
sentry-cli releases new        "$RELEASE"
sentry-cli releases set-commits --auto "$RELEASE"
sentry-cli releases deploys    "$RELEASE" new -e $CHANNEL
```
- Android：上传 ProGuard mapping + `.so` (含 debuginfo) → Sentry 自动抽 buildid。
- iOS：上传 `.dSYM`（zip）。
- Windows：上传 `.pdb`。
- Linux：上传带 debug 的 `.so` 或剥离后的 `.dbg`，并保留 `build-id`。

符号文件 **绝不入** SDK 包（防止泄漏）；包里只放剥离后的二进制。

### 4.2 MinIO 上传（构建产物）

构建产物 zip 包通过 `upload.ps1` 直传 MinIO，存储路径按产品、分支、平台、版本、日期（精度到天）分层：

```
s3://sdk-packets/{product}/{branch}/{platform}/{version}/{date}/{zip_name}
```

**分支提取规则**（由 `upload.ps1` 从版本号解析）：
| 版本号模式 | 分支 |
|---|---|
| `*-develop.*` | `develop` |
| `*-hotfix.*` | `hotfix` |
| `*-rc.*` 或 `*-staging.*` | `rc` / `staging` |
| 其他（如 `1.0.0`） | `release` |

**日期提取**：从 zip 文件名中的 `YYYYMMDD` 时间戳段提取（精度到天）。若文件名不含时间戳，则使用当前日期。

**上传幂等性**：上传前先 HEAD 检查对象是否已存在，已存在则跳过，不会覆盖。由于文件名包含秒级时间戳 + 构建号 + commit hash，不同构建产生同名文件的概率近似为 0。

**上传流程**：
1. HEAD 请求检查对象是否存在
2. 若 200 → 跳过（已存在）
3. 若 404 → 执行 PUT 上传
4. 上传后 GET 下载前 16 字节验证完整性
5. 成功后删除本地 zip 文件（可选，通过 `-DeleteAfterUpload` 开关控制）

### 4.3 符号表镜像备份：MinIO（可选）

> ⚠️ 当前 demo 未启用符号表 MinIO 镜像备份，符号表仅走 Sentry 一条线。

若需冗余备份，可在构建末尾把 `symbols/` 打 `zst` 上传至 `s3://sdk-packets/{product}/{branch}/{platform}/{version}/symbols/`。意义：
- Sentry 出问题时本地仍能用 `minidump_stackwalk` / `lldb` 手工符号化。
- 离职/合规审计需要原始符号文件。

### 4.4 老版本兼容

线上 release 的符号表 **永久保留**。Sentry self-host 的 PostgreSQL/ClickHouse 体量主要在 events 而不是 symbols，几年内不用担心。

## 5. crash 与版本对应

- SDK 初始化时把 `release = {product}@{version}-{buildId}`、`environment = {channel}`、`dist = {buildId}` 注入 Sentry SDK（或自家上报）。
- 用户上传的 raw dump 也会带 `meta.json` 里写 `release` 字段（见 06 章），decrypt-proxy 解析 meta.json 后可一键关联 Sentry issue。

## 6. SDK 接入方拉包路径

| 平台 | 用法 |
| --- | --- |
| Android | `repositories { maven { url 'https://nexus.intra/repository/sdk-maven-release/' } }` → `implementation 'com.mycompany.rtc:sdk:1.7.0'` |
| iOS | `pod 'RTCSDK', '1.7.0', :source => 'git@git.intra:sdk/cocoapods-spec.git'` |
| Windows / Linux | 直接 `curl` Nexus raw 或在 CMake 里 `FetchContent` |
| 内部开发 dev 包 | 把 url 换成 `sdk-maven-dev` / `sdk-raw-dev` 即可 |

通过 Nexus group 仓 `sdk-group` 把 release/staging/dev 聚合，**开发者只需切一个 profile 参数**。

## 7. 提测 / 发布闭环

```
开发者:  push 分支
        ↓
Jenkins: dev 通道出包 (自动) → 飞书通知 + 提供 dev 下载链接
        ↓
QA 准备: 开发提"提测"PR → 合到 release/x.y 或打 vX.Y.Z-rc.N
        ↓
Jenkins: staging 通道, input 弹窗 → QA leader + 技术 leader 双确认 → 写 sdk-raw-staging
        ↓
QA 验完: release manager 打 v1.7.0 tag (gpg signed)
        ↓
Jenkins: release pipeline, 强校验:
  - tag 必须 gpg signed 且签名人在白名单
  - staging 必须存在同 commit 的包 (防止"重新编译")
  - 强制 ALLOW_ONCE
  - input 二次审批
        ↓
写 sdk-raw-release / sdk-maven-release / sdk-cocoapods-release
        ↓
通知 + 自动更新 changelog 网站
```

## 8. 元数据后端（可选轻量）

如果想在 Web 上做"全文检索版本"，加一个 SQLite/Postgres 表 `builds`，字段 = `metadata.json` 扁平化，由 publish.sh 末尾 `curl` 一次后端 `POST /builds` 写入。前端 Web 查询非常快。

> 注意：**Nexus + Sentry 已经能查 90% 场景**，这个表只是给"全局聚合视图"用，不是必需。

## 9. 一句话总结

> **三通道 + 不可变版本 + manifest + Sentry release**，crash、符号、源码、构建机器一条龙能反查。Nexus 管包，Sentry 管符号与崩溃，MinIO 做符号冷备 —— 各司其职，运维只盯三件事。
