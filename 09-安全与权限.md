# 09 安全与权限

> 一句话：**SSO 统一登录 + 三组件 RBAC + 客户端无长期凭证 + 解密通道唯一持密钥 + 全链路审计**。

## 1. 身份认证（SSO）

- 推荐用 **Gitea 自带 OAuth2/OIDC Provider**（小团队最省事，账号即 Git 账号）。
- 大一点可放 **Authentik / Keycloak**。
- 接入对象：
  - Jenkins（OIDC plugin）
  - Nexus（OIDC plugin / SAML）
  - Sentry（OIDC / OAuth）
  - MinIO（OIDC）
  - Filebrowser（OIDC）
  - **decrypt-proxy**（OIDC，必须）
  - Caddy（forward_auth 作为兜底）

## 2. 角色（RBAC，跨组件对齐）

| 角色 | Git | Jenkins | Nexus | Sentry | MinIO | decrypt-proxy |
| --- | --- | --- | --- | --- | --- | --- |
| `developer`        | 写自己仓库     | 跑 dev/staging job | 拉所有，写 dev | 读自己 project | 读 dev/staging | 看 + 下载 dev/staging 明文 |
| `qa`               | 只读           | 跑 staging job、审批提测 | 拉所有，写 staging | 读所有 | 读 staging | 看 + 下载 staging 明文 |
| `release-manager`  | 写 release/*   | 审批 release | 写 release（ALLOW_ONCE） | 读所有 | 读所有 | 看 + 下载 release 明文 |
| `sre / admin`      | admin          | admin | admin | admin | admin | admin（管 keyring/审计） |
| `external-customer`| —              | — | 仅拉 release | — | — | — |

> 角色挂在 SSO 组里（`group: developers/qa/release-managers/sre`），各组件读 group claim 完成授权，**只在一个地方加人**。

## 3. 网络分区

```
[公司内网]                              [DMZ / 公网?]
   开发者浏览器 ────► Caddy (TLS) ───► 各服务容器内部
                       │
                       └── OIDC → Gitea

[客户终端 (外网)]
   APP ───► token.intra (TLS) ──► upload-token-service
        ──► s3.intra (TLS, presigned PUT) ──► MinIO
```

- 内部服务（Jenkins、Nexus、Sentry、Filebrowser、MinIO Console、decrypt-proxy）**只对内网**或 VPN 暴露。
- 对外暴露的只有 `token.intra` 和 `s3.intra`（且 s3.intra 只接受带签名的 PUT/GET）。
- MinIO 的 admin 接口 9001 绝对不开公网；S3 接口 9000 通过 Caddy 反代 + WAF 规则。

## 4. 凭证管理

| 凭证 | 哪里存 | 谁能拿 |
| --- | --- | --- |
| 客户端 device JWT | OS keystore | 仅 SDK 进程 |
| 客户端 master key 派生材料 | OS keystore | 仅 SDK 进程 |
| MinIO 后台 root | `.env` / Vault | sre |
| MinIO 服务账号（jenkins/uploader/proxy） | Jenkins credentials / proxy env | 对应进程 |
| Nexus CI 账号 | Jenkins credentials | Jenkins |
| Sentry token | Jenkins credentials | Jenkins |
| Gitea webhook secret | Jenkins credentials | Jenkins |
| **decrypt-proxy keyring** | `/srv/.../keyring.json` 600 + Vault Agent | 仅 proxy 进程用户 |
| GPG 签名 release tag 私钥 | release-manager 个人 YubiKey | 仅本人 |
| 代码签名证书（Win/Apple） | macOS Keychain / Windows cert store on build agent | 仅该 agent |

强烈建议放 **HashiCorp Vault**（OSS 单节点足够小团队），所有服务用 `vault agent` 拉，不在 git 里出现明文。如果嫌重，至少做 `sops + age` 加密。

## 5. 客户端安全

- **不下发 S3 access key**：用 presigned URL，TTL 5 分钟。
- **不下发 master key**：用 HKDF 派生 + OS keystore；丢失设备无法被复用。
- **device_id 防伪造**：activate 时校验 app 签名 + 包名 + 防 root/debug；服务端做 IP/速率风控。
- **完整性**：上传 sha256 双向校验，objects bucket 开 versioning（防误覆盖，配合 lifecycle 控容量）。
- **不上传 PII**：日志 SDK 在客户端强制脱敏（手机号、token、url query 等用正则掩码），评审 once 后归入 SDK 内部测试。

## 6. 服务端硬化

- portal 主机：
  - 关闭 root SSH，仅 key 登录。
  - ufw / firewalld 仅放 22/80/443。
  - docker rootless 或限制 capabilities。
  - 自动化 patch：unattended-upgrades（Debian）/ dnf-automatic（RHEL）。
- 容器：
  - 自有 `sdk-docker` registry 镜像每周扫描（trivy）。
  - 容器 read-only filesystem，写卷只挂数据目录。
- TLS：Caddy 自动续签，最低 TLS1.2；内部服务也强制 TLS（mTLS 可选 between proxy ↔ MinIO）。

## 7. 审计 / 日志

| 系统 | 关键事件 |
| --- | --- |
| Jenkins         | login, job start/finish, input approval（含 submitter） |
| Nexus           | upload/download/delete（带 user/repo） |
| Sentry          | release create, debug-file upload, user invite |
| MinIO           | bucket notification → `audit/` topic，包含 GET/PUT/DELETE |
| decrypt-proxy   | login, search, download_raw, download_plain, symbolicate, keyring change（JSONL） |
| Caddy           | access log（敏感字段不记） |

集中归集：把所有 json 日志通过 Vector / Fluent Bit 推到 Loki / OpenSearch，前置 Grafana 看；保留 1 年。

## 8. 合规与法务最小要求

- 所有 release 的 SDK 包内置 LICENSE + SBOM（`syft -o cyclonedx-json`）。
- 用户协议中说明日志/dump 上传内容、范围、保留期、删除请求接收点。
- GDPR/个保法配套：在 decrypt-proxy 的 `/admin/erase` 提供按 `device_id` 删除（MinIO 批量删 + catalog 标记 erased）。

## 9. 灾备演练

每季度跑一次：
1. portal 主机模拟宕机，从异机房恢复。
2. 模拟 keyring.json 丢失 → 走密封信封流程（Shamir secret sharing 3-of-5）。
3. 模拟一次 release 包被误删 → 从备份恢复，验证 sha 与 sentry release 关联仍正常。

## 10. 一句话总结

> **凭证最小化、密钥单点持有、SSO 统一、RBAC 对齐、链路全审计**。中小团队只用三件事就够：Vault 管秘、Gitea 管身份、Loki 管日志。
