# 08 保留与清理策略

> 一句话：**两类对象（SDK 包 / 日志 dump）按通道差异化保留**，由 Nexus Cleanup Policy + MinIO Lifecycle 自动执行，不依赖 cron 脚本。

## 1. SDK 包（Nexus）

| 通道 | 保留策略 | 实现 |
| --- | --- | --- |
| `sdk-raw-dev` / `sdk-maven-dev`         | 保留最近 30 天 **或** 最新 20 个版本，取较大者 | Nexus Cleanup Policies → 按 `lastDownloaded > 30d AND lastModified > 30d`；keep last N 由 `keep` task 实现 |
| `sdk-raw-staging` / `sdk-maven-staging` | 6 个月                                                                             | 同上，`> 180d` |
| `sdk-raw-release` 等                    | **永不自动删**                                                                     | 不配 cleanup |
| Docker 镜像 `sdk-docker`                 | 保留 last 30 标签，且 `release/*` tag 永不删                                       | Nexus image cleanup |

在 Nexus 里：System → Cleanup Policies → 新建：
- `cleanup-dev-30d`：format=raw, criteria=lastBlobUpdated>30 days
- `cleanup-staging-180d`：同理
- 绑定到对应仓库，启用每日 02:00 跑 `Cleanup unused tasks`。

## 2. 日志 / Dump（MinIO 生命周期）

MinIO 与 S3 完全兼容 lifecycle，按 **对象 Tag** 触发。S3 Tag 由客户端上传时通过 `x-amz-tagging` 头带上，或由 catalog ingest 服务在 `ack` 阶段补打（推荐后者，更可控）。

`s3://sdk-logs/` Lifecycle：

```xml
<LifecycleConfiguration>
  <Rule>
    <ID>logs-dev-14d</ID>
    <Filter><Tag><Key>channel</Key><Value>dev</Value></Tag></Filter>
    <Status>Enabled</Status>
    <Expiration><Days>14</Days></Expiration>
  </Rule>
  <Rule>
    <ID>logs-staging-90d</ID>
    <Filter><Tag><Key>channel</Key><Value>staging</Value></Tag></Filter>
    <Status>Enabled</Status>
    <Expiration><Days>90</Days></Expiration>
  </Rule>
  <Rule>
    <ID>logs-release-180d</ID>
    <Filter><Tag><Key>channel</Key><Value>release</Value></Tag></Filter>
    <Status>Enabled</Status>
    <Expiration><Days>180</Days></Expiration>
  </Rule>
</LifecycleConfiguration>
```

`s3://sdk-dumps/`：

| Channel | 保留 | 备注 |
| --- | --- | --- |
| dev     | 30 天 | |
| staging | 180 天 | |
| release | **永久**（或归档冷存） | 关键调查证据，**严格保留** |

`s3://sdk-symbols/`：与对应 SDK 通道保留同步；release 永久。

> 把策略写成 IaC：`infra/minio/lifecycles/*.json`，用 `mc ilm import` 在部署时一次性同步。

## 3. catalog DB 与对象的"软删除一致性"

对象被 lifecycle 删后，catalog 表行仍在。两种做法：

1. **保留行**（推荐）：catalog 加字段 `deleted_at`；用 MinIO bucket notification → webhook `decrypt-proxy /events/object-removed` 标记 `deleted_at = now()`。Web 列表默认隐藏，给"显示已过期"勾选。
2. **同步删除**：定时跑 `cleanup.py`，比对 catalog vs MinIO，列差集删行。

第 1 种保留了"曾经存在过这个 dump"的事实，便于做趋势统计。

## 4. 异地备份

| 数据 | 备份目标 | 频率 |
| --- | --- | --- |
| `s3://sdk-logs/`、`sdk-dumps/` 上 release 标签的对象 | 异机房 MinIO / 云 OSS | `mc mirror --watch` 实时 |
| `s3://sdk-symbols/release/*` | 同上 | 实时 |
| Nexus blobstore | tar + zstd 上传 MinIO `s3://backup/nexus/yyyy-mm-dd.tar.zst` | 每日 |
| Sentry Postgres | `pg_dump` → MinIO `s3://backup/sentry/` | 每日 |
| Jenkins `JENKINS_HOME` | tar 上传 MinIO | 每日 |
| catalog DB（Postgres） | `pg_dump` → MinIO | 每小时 |

实现：`scripts-stub/backup-*.sh` + systemd timer。

## 5. 容量监控

- Prometheus + node_exporter + minio_exporter + postgres_exporter。
- Grafana 看板预警：
  - portal 数据盘 > 75% 黄、> 90% 红。
  - MinIO bucket size 增速 > 20%/周。
  - Nexus blob `release/` 桶单独看（永不清，必须有冷却容量计划）。
  - Sentry events 量超 plan。

阈值红色时自动给值班飞书发卡。

## 6. 删除审计

任何对 release 通道对象的人工删除都必须：
1. 工单先行（任何 IM 都行）。
2. 由 admin 角色在 decrypt-proxy `/admin/delete` 操作（写 audit）。
3. MinIO 端打开 **Object Locking + Compliance mode**（release bucket 也可开 Governance mode，给定时间窗内禁止删）。

## 7. 一句话总结

> **dev 短保、staging 半年、release 永久**；自动化交给 Nexus Cleanup + MinIO Lifecycle；catalog 软删，事实可追；异地备份脚本化，监控配齐。
